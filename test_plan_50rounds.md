# 社區管理系統 50 輪企業級 E2E 測試計畫
## 聚焦：訪客管理 + 包裹管理 (跨模組交互)

> 測試環境：Odoo 18 CE @ localhost:9098
> 帳號：admin/admin (後台), portal/portal (住戶 王大明), 外部訪客 (公開連結)
> 工具：playwright-cli + named sessions (-s=admin, -s=portal, -s=visitor)

---

## Phase 1: 訪客管理核心流程 (R1-R10)

### R1: 臨時訪客完整生命週期 — 從登記到離場
- **角色**: admin (警衛)
- **步驟**: 建立新訪客 → 建立來訪記錄(臨時) → 發送確認 → 住戶確認 → 報到 → 離場
- **驗證**: 狀態機完整轉換 draft→pending→confirmed→checked_in→checked_out
- **狀態**: pending

### R2: 住戶透過 Portal 確認訪客
- **角色**: portal (王大明)
- **步驟**: 登入 Portal → 查看待確認訪客列表 → 點擊確認放行
- **驗證**: 訪客狀態變為已確認, 住戶確認時間記錄
- **狀態**: pending

### R3: 住戶透過 Portal 拒絕訪客
- **角色**: admin + portal
- **步驟**: 建立訪客來訪 → 發送確認 → 住戶拒絕
- **驗證**: 狀態變為已拒絕, 拒絕住戶記錄
- **狀態**: pending

### R4: 預約通行單建立與驗證碼核銷
- **角色**: portal (建立) + admin (核銷)
- **步驟**: Portal 建立預約 → 後台驗證碼查驗 → 自動入場
- **驗證**: 預約建立、驗證碼產生、QR Code、入場記錄建立
- **狀態**: pending

### R5: 週期性預約 — 工作日定期訪客
- **角色**: portal + admin
- **步驟**: 建立週期性預約(週一至五 09:00-18:00) → 驗證在授權時段內核銷
- **驗證**: 週期性時間檢查邏輯、剩餘次數
- **狀態**: pending

### R6: 訪客黑名單機制
- **角色**: admin
- **步驟**: 將訪客加入黑名單 → 嘗試用該訪客的預約通行單核銷 → 預期失敗
- **驗證**: 黑名單標記、核銷拒絕訊息、黑名單日期
- **狀態**: pending

### R7: 訪客證管理 — 發放與回收
- **角色**: admin
- **步驟**: 查看訪客證列表 → 入場時發放訪客證 → 離場時自動回收
- **驗證**: 訪客證狀態 available→in_use→available
- **狀態**: pending

### R8: Portal 預約管理 — 查看、建立、取消
- **角色**: portal
- **步驟**: 查看預約列表 → 建立新預約 → 查看詳情(QR) → 取消預約
- **驗證**: Portal CRUD 功能完整性
- **狀態**: pending

### R9: 後台訪問記錄篩選與搜尋
- **角色**: admin
- **步驟**: 今日篩選 → 狀態篩選 → 類型篩選 → 戶號分組 → 搜尋特定訪客
- **驗證**: 搜尋篩選功能完整性、預設篩選(今日)
- **狀態**: pending

### R10: 確認逾時自動處理
- **角色**: admin
- **步驟**: 建立訪問 → 發送確認 → 不操作等待逾時 → 驗證逾時狀態
- **驗證**: Token 過期機制、狀態變為 timeout
- **狀態**: pending

## 測試標準

### 商用企業部署要求
1. **功能正確性**: 所有狀態轉換符合預期
2. **資料完整性**: 序號、時間戳、關聯記錄正確
3. **安全性**: 權限隔離、跨戶保護
4. **穩定性**: 重複操作不出錯
5. **使用者體驗**: 操作邏輯合理、回饋即時
6. **邊緣案例**: 異常輸入不崩潰

### 測試員角色定義
- **QA Lead**: 獨立測試員視角，以真實社區管理員/住戶身份操作
- **評估標準**: 現實世界場景合理性、操作順暢度、錯誤處理品質
